# 说人话：宣传与冷启动方案

> 适用目标：你已经在 X（推特）和 Linux.do 发过首波内容，但后续曝光没有接上。  
> 这份方案的目标不是“立刻爆”，而是帮你把这个项目从一次性发帖，变成一个有节奏的冷启动流程。

---

## 1. 先说结论

你现在最缺的不是“再发一次链接”，而是 **第二波叙事素材**。

单纯转发仓库链接，用户很难理解：

- 它和普通 prompt 有什么不同
- 为什么中文场景需要这个 skill
- 它具体解决了什么痛点
- 为什么不是又一个“改写一下文案”的工具

所以第二轮曝光前，先补这 4 个素材：

1. **更专业的 README**
2. **至少 3 组能秒懂的 before / after**
3. **一条征集 bad cases 的入口**
4. **一个明确的版本升级点（比如 v1.7 要做 voice / 二次审稿）**

没有这 4 个，发帖只会继续变成“有人看了，觉得有点意思，但不知道接下来要不要关注”。

---

## 2. 这轮宣传的目标

不要只盯 star。更现实的 30 天目标应该是：

### 质量目标

- 拿到 20–30 条真实 bad cases
- 收到 5–10 条用户真实使用反馈
- 积累 5 条以上可公开展示的真实 before / after

### 分发目标

- GitHub README 完成重写
- 至少被 2–4 个 skill / agent 目录收录
- 完成第二轮 X 帖子和 Linux.do 跟进帖
- 建立一个长期可复用的 launch SOP

### 数据目标（可选）

如果你想设数字，建议保守一点：

- star：先看能不能多拿 50–150 个高质量关注
- issues / discussions：至少有 5 个真实样本反馈
- 外部目录 / 列表：2–4 个

---

## 3. 核心传播角度

这轮传播不要再泛泛地说“去 AI 味”。  
要把项目说成一个 **有明确病灶、有边界、有 benchmark 的中文 skill**。

## 3.1 推荐的 3 条传播主线

### 主线 A：中文 AI 写作有自己独特的病灶

重点讲：

- 不只是 `赋能 / 闭环`
- 现在还有 `收窄 / 坐实 / 兜住 / 收口` 这种工程师表演腔
- 还有 `保姆级 / 绝绝子 / 建议收藏` 这种小红书 AI 腔
- 英文项目很少专门解决中文这些问题

这条线适合 X 和 Linux.do。

### 主线 B：这不是乱润色，它有边界和误杀防护

重点讲：

- docs / status / code-context 默认保守
- 不乱改代码、命令、参数、日志
- benchmark 分 should-fix 和 should-not-fix
- 不只是“改得更好看”，而是“该改的改，不该改的别碰”

这条线适合 GitHub README、Release、目录提交。

### 主线 C：这个项目下一步不是堆词表，而是做 voice / 二次审稿

重点讲：

- 当前版本已经能做基础清理
- 下一步要解决“改完还是有轻微 AI 味”
- 方向是 voice calibration、residual audit、真实样本评测

这条线适合开发日志、跟进帖、征集反馈。


## 4.2 演示素材

至少准备 3 组：

### 素材 1：最能打的 before / after

拿一段典型“AI 味重”的中文段落，展示改前改后。

### 素材 2：工程师腔案例

这是你的辨识度之一。  
把 `收窄 / 根因 / 兜住 / 收口` 这类案例做成一组展示。

### 素材 3：annotation mode

很多人不想一上来就让 AI 改稿，先看问题点会更安心。

> 重点：不要只有链接，要有看一眼就懂的截图/对照内容。

## 5. 渠道优先级

## P0：GitHub 自身

最先优化的不是外部平台，而是仓库本身。因为所有外部流量最后都会落到这里。

要做的事：

- README 重写
- release note 更专业
- pinned issue / discussion
- topics 更新
- examples 更可读

## P0：X（推特）

你已经发过一次，第二波不要再发“项目发布了”。  
要发 **洞察 + 案例 + 请求反馈**。

推荐发 3 条，而不是 1 条：

1. **洞察帖**：中文 AI 写作里最烦的几个新病灶
2. **案例帖**：一个 before / after，最好是工程师腔
3. **征集帖**：欢迎贴“改完还是像 AI”的案例

## P0：Linux.do

Linux.do 适合发长一点的“做这个项目时我观察到了什么”。

比起单纯发链接，更适合这种结构：

1. 我为什么做这个项目
2. 我发现了哪些中文 AI 味新症状
3. 为什么我没有把它做成敏感词替换器
4. benchmark 怎么设计
5. 目前哪里还不够好
6. 欢迎大家贴 bad case

这样比“项目地址在这里，欢迎 star”更容易带来真实反馈。

## P1：技能目录 / 合集

建议优先提以下几个方向：

- `awesome-agent-skills` 类仓库
- agent skill directory / registry
- 适合 Codex / Claude / Copilot 的技能清单
- 可提交 skill 的目录网站

### 你这轮最值得做的，不是广撒网，而是先拿下 2–4 个质量高的目录

这样有两个好处：

- 项目会有稳定的长尾入口
- 以后你做下一个 skill，也知道该提到哪里

## P1：开发者社区

等 README 和英文说明更稳之后，再考虑：

- V2EX
- 掘金
- Reddit 的相关子版块
- GitHub 相关讨论区

## 6. 14 天动作表

## Day 1–2：补基础面

- 替换 README
- 补一个 pinned issue / discussion：征集 bad cases
- 准备 3 组 before / after
- 补 repo topics
- 确定下一版本路线：voice + residual audit

## Day 3–4：做目录提交

- 选 2–4 个 skill / agent 目录
- 准备项目简介、标签、安装方式、示例
- 发 PR 或提交表单

## Day 5：发 X 洞察帖

主题建议：

> “中文 AI 写作最近最烦的，不只是赋能闭环，而是开始说‘收窄、坐实、兜住、收口’了。”

结构：

1. 现象
2. 一个案例
3. 项目链接
4. 征集 bad cases

## Day 6–7：发 Linux.do 跟进帖

不要重复首发，改成：

- 这几天我收到的反馈
- 我发现这个 skill 目前最大的短板
- 我准备怎么改（voice / 二次审稿）
- 欢迎继续贴案例

## Day 8：发第二条 X 案例帖

拿最强的一组 before / after，直接展示结果，不讲太多背景。

## Day 10：发一个小版本或开发日志

哪怕不是正式 release，也可以发 devlog：

- 我补了什么规则
- 我发现哪里还不稳
- 我现在重点解决什么问题

## Day 12：发“求 bad cases”帖

最有效的传播，不一定是展示你有多强，而是展示你在认真做边界和反馈闭环。

## Day 14：做一条阶段总结

内容包括：

- 这两周拿到了多少反馈
- 哪些 case 最难
- 接下来要做什么
- 感谢谁提供了样本

## 9. 这轮宣传最容易踩的坑

## 坑 1：只发“项目地址”

没人有义务自己理解你的项目。  
必须给一个最短路径的案例。

## 坑 2：只谈“去 AI 味”，不谈边界

会让人误以为这是普通 prompt 或 detector-bypass 工具。

## 坑 3：只有技术亮点，没有反馈入口

如果没有“欢迎贴案例”的入口，曝光过去了也不会形成反馈闭环。

## 坑 4：一次性把所有话说完

不要一条贴子把所有信息塞满。  
更好的节奏是：

- 第一条：洞察
- 第二条：案例
- 第三条：征集反馈
- 第四条：开发进展

---

## 10. 你这次最值得复用的 launch 公式

以后你做别的项目，也建议继续用下面这个公式：

### 一句话定位
这个项目是干什么的，解决什么具体痛点。

### 一个最强案例
不要只讲功能，先给 before / after。

### 一个边界声明
明确说明它不是做什么的。

### 一个反馈入口
给别人一个最容易参与进来的动作。

### 一个下一步
告诉别人“项目不是发完就没了”。

---

## 11. 最后的宣传判断

`说人话` 这种项目不太适合靠一次性大曝光冲起来。  
它更适合的是：

> **用洞察吸引人，用案例让人看懂，用边界建立信任，用 bad case 收集推动下一轮迭代。**

你下一轮最重要的目标，不是“再发一次”，而是把这个项目的传播节奏变成：

- 有内容
- 有反馈
- 有版本
- 有继续关注的理由

只要这件事跑通，后面你做别的项目，流程也就跑通了。
