# 说人话：产品迭代文档（vNext）

> 版本日期：2026-04-12  
> 适用对象：项目作者、未来协作者、Codex  
> 文档目的：把 `说人话` 从“好玩的去 AI 味 skill”推进成“可持续迭代、可评测、可传播的中文自然表达协议”

---

## 1. 文档目标

这份文档要解决 4 个问题：

1. **项目下一步往哪走**
2. **为什么现在的效果还不够稳，偶尔还是有 AI 味**
3. **怎样把需求写清楚，让 Codex 能继续接着开发**
4. **怎样让这个项目不只是一个规则仓库，而是一个可长期维护的产品雏形**

---

## 2. 当前项目画像

### 2.1 当前它已经做得不错的地方

`说人话` 已经不是一个“随便列几个敏感词替换”的 prompt 了，而是一套有边界、有 benchmark、有误杀防护的 rewrite skill。

当前仓库已经具备这些基础能力：

- 中文优先，同时兼顾英文
- 210+ 中文短语、96 英文短语、19 类结构反模式
- 场景路由：`chat / status / docs / public-writing`
- 强度控制：`minimal / standard / aggressive`
- `annotation mode`：只标问题，不直接改稿
- 无源引用策略：`rewrite-safe / audit-only / rewrite-with-placeholder`
- benchmark 与误杀防护意识
- 安装文档和 changelog 已经成型

这说明项目不是“点子阶段”，而是已经进入了 **可产品化的早期阶段**。

### 2.2 当前最强的产品资产

目前最值钱的，不是词表，而是这三样：

1. **你对中文 AI 写作病灶的观察是有辨识度的**
   - 不只是“赋能、闭环”这种老词
   - 还抓到了“工程师腔 / 调试腔 / 小红书 AI 腔 / narrator 腔”这类更有传播力的现象

2. **你有 benchmark 思维**
   - 很多同类项目只会不停加 prompt
   - 你已经把“该改的要改，不该改的别误杀”写成评测资产

3. **你不是只追求更像人，而是强调保事实、保术语、保责任主体**
   - 这决定了它更像一个 serious tool，而不是花活 prompt

### 2.3 当前最现实的问题

你已经明确反馈：**实际用起来，效果还没有达到预期，还是会有 AI 味。**

这不是偶然，而是现在这类规则型 skill 的典型瓶颈。当前版本更擅长：

- 删套话
- 去表演感
- 控制误杀
- 守住场景边界

但还不够擅长：

- 让文本真的“像某个具体人在说话”
- 在保守场景里做到“不油但不僵”
- 第一轮清理后继续发现残留味道
- 在真实发布文本里做到“能直接发”

## 3. 核心问题定义

### 3.1 用户真正要的不是“删词”，而是“像人”

用户嘴上说“去 AI 味”，实际要的通常是下面四件事的组合：

- 读起来不要像模板
- 不要像模型在表演理解力和执行力
- 不要把正常信息写成套话
- 最好还有点“像我本人会说的话”

所以 `说人话` 的目标不该停在“AI 套路清理器”，而应该升级为：

> **中文自然表达协议：在不改事实和术语的前提下，把模型输出拉回当前场景里更自然、更可发布的人类表达。**

### 3.2 当前版本为什么还会“有 AI 味”

#### 问题 1：负向规则强，正向风格弱

现在的仓库非常擅长告诉模型：

- 不要说什么
- 哪些话别写
- 哪些结构有问题
- 什么场景不要乱动

但对模型来说，只靠“不能说什么”通常只能得到 **更干净的文本**，不一定能得到 **更像真人的文本**。

典型结果是：

- 套话没了
- 黑话少了
- 信息没丢
- 但还是平、还是整齐、还是像“被清理过的模型文本”

#### 问题 2：缺少“声音锚点”

现在的 `说人话` 更像一个清洁协议，不像一个风格适配协议。

也就是说：

- 它会把“AI 味”去掉一部分
- 但不会主动学“这个用户平时怎么说话”
- 也不会根据样本建立“这个场景里允许什么程度的口语、判断、节奏变化”

没有声音锚点，就容易出现“干净但没个性”。

#### 问题 3：评测更偏规则，不偏结果体验

现在 benchmark 很强，但主要回答的是：

- 该改的有没有改
- 不该改的有没有误杀

它还没有系统回答：

- 改完以后像不像真人
- 像不像用户自己
- 是否可以直接发
- 是否过于平滑、过于工整、过于解释型

这会导致“规则正确”与“体验满意”之间有空档。

#### 问题 4：单文件模式和完整模式的差距，没有被产品化表达清楚

当前文档里已经提到：

- `SKILL.md` 是兜底
- `references/` 才是更完整的模式

但对普通用户来说，这种差别还不够“产品化”。结果就是：

- 用户以为只装 `SKILL.md` 就等于全部能力
- 一旦效果一般，就会觉得项目本身不够好
- 实际上是使用模式和预期没对齐

#### 问题 5：缺少“残留味二次审稿”

很多文本不是第一遍改写完就自然了。常见情况是：

- 第一遍删掉大块套话
- 第二遍才会发现剩下的句子还是太匀、太顺、太像在交作业

如果没有“rewrite 之后再做一次 residual audit”，结果往往是 **从重 AI 味变成轻 AI 味**，而不是变成真自然。

#### 问题 6：没有“真实样本产品化”

你现在的样本更偏 benchmark 和规则验证，非常好，但还不够像真实作者会拿来直接发的那类材料，比如：

- X / Linux.do / V2EX 帖子
- 项目 README 段落
- release note
- commit message
- issue 回复
- 开发者日常说明

未来如果不把这些样本沉淀成资产，项目就会一直停留在“看起来很强”，而不是“用起来真顺”。

## 4. 产品定位升级

## 4.1 建议的新定位

### 一句话定位

**让 AI 写出来的中文，少一点模板感和表演感，多一点正常人会在当前场景下说的话。**

### 面向 GitHub / 开源社区的定位

**A Chinese-first rewrite skill for removing AI writing patterns while preserving facts, terminology, and technical context.**

### 面向用户的价值表达

不是“替换敏感词”，不是“润色到更华丽”，而是：

- 去模板感
- 去表演感
- 去语域漂移
- 保事实
- 保术语
- 保可发布性

## 4.2 产品边界

### 做什么

- 面向中文优先场景的自然表达重写
- 处理 AI 套话、黑话、翻译腔、工程师表演腔、过度总结腔
- 在不同场景下控制改写强度
- 支持先诊断再改写
- 支持保守模式，避免乱改 docs / status / code-context

### 不做什么

- 不以“骗 AI detector”为目标
- 不凭空补数据、来源、细节
- 不模仿名人或未经授权的品牌 voice
- 不把严肃技术文档硬改成口语帖
- 不把所有文本都改成一种统一“互联网口吻”

## 5. 核心设计原则

### 原则 1：先保真，再自然

任何自然表达能力，都不能以牺牲事实、术语、责任主体为代价。

### 原则 2：先做模式识别，再做词表补丁

不要掉进“越不好用越加词”的陷阱。词表可以扩，但真正决定上限的是：

- 模式归类能力
- 场景判断能力
- 二次审稿能力
- 真实样本校准能力

### 原则 3：从“负向约束”升级到“负向约束 + 正向风格目标”

负向约束负责 **去掉 AI 味**，正向风格目标负责 **避免改完以后像被消毒过的文本**。

### 原则 4：把“完整模式”产品化

必须让用户清楚知道：

- `SKILL.md` 是兜底
- `references/` 是精细模式
- 某些场景只靠单文件不会有最佳效果

### 原则 5：所有新能力都必须可评测

不接受“感觉更好了”这种迭代。每次新增能力都要同时新增：

- 对应用例
- 边界样本
- README 说明
- CHANGELOG 记录

## 6. 目标用户与 JTBD

## 6.1 核心用户

### 用户 A：独立开发者 / AI 重度用户

特点：

- 经常让 Codex / Claude / ChatGPT 帮写 README、项目介绍、帖子、注释
- 最怕结果“看着还行，但一眼 AI”
- 愿意装 skill，但不愿意搞特别复杂的配置

JTBD：

> “我想保留 AI 带来的效率，但不要让最终文本看起来像模型写的。”

### 用户 B：技术写作者 / 开源作者

特点：

- 会写 release note、技术博客、公开帖子
- 需要专业感，但不想像 PR 文案
- 会介意术语被乱改

JTBD：

> “我想让文本更自然，但不要失真，也不要被过度口语化。”

### 用户 C：审稿者 / review 角色

特点：

- 不一定想让 AI 直接重写
- 更想先知道哪里有问题
- 需要保守地指出风险，不主动改坏

JTBD：

> “先告诉我哪里不自然、哪里缺来源、哪里像模板，我再决定怎么改。”

### 用户 D：希望沉淀自己 voice 的用户

特点：

- 已经不满足于“去 AI 味”
- 希望结果开始接近自己平时的表达

JTBD：

> “我不只要更自然，我还要更像我自己会写的东西。”

## 7. 产品方案：从 skill 升级为协议

未来的 `说人话` 不应该只有“检测+改写”，而应该形成一个更完整的协议栈。

## 7.1 建议的能力架构

### 1）Scene Router

先判断主场景和保守边界：

- `chat`
- `status`
- `docs`
- `public-writing`
- 后续可细分：`forum-post / X-post / README / release-note / commit-message / docstring`

### 2）Protected Spans

先抽取不该乱动的内容：

- 数字
- 时间
- 专有名词
- 引用
- 命令
- 代码
- 路径
- 参数
- 报错
- API 名

### 3）Issue Detector

识别问题族，而不是只查词：

- 开场套话
- 总结式收尾
- 工程师表演腔
- 商业黑话
- narrator 腔
- 语域混搭
- 价值拔高骨架
- 无源引用
- 节奏过匀

### 4）Positive Style Contract

给模型明确的正向目标，例如：

- 更具体
- 更少空洞判断
- 长短句稍微有变化
- 少一点“结论先说”“解释自己在解释”
- 保留正常人会有的轻微不对称，而不是把每句都抛光到一样整齐

### 5）Voice Calibration Lite

允许用户提供 2–3 段自己的文字样本，提取：

- 句长节奏
- 口语度
- 标点习惯
- 连接方式
- 主语偏好
- 语气强弱

注意：这是“拟合用户自己提供的 voice”，不是仿别人。

### 6）Residual Audit

第一遍改写后，进行一次极短的二次审稿：

- 有没有残留开场/总结腔
- 有没有过匀节奏
- 有没有空泛判断
- 有没有把文本改得太整、太像模板答案

### 7）Output Contract

对 agent/工具给出可复用输出格式。默认给成稿；需要时支持诊断结构化输出。

## 8. 版本路线图

## 8.1 v1.7：从“删 AI 味”到“更像真人”

### 目标

解决你当前最真实的抱怨：**清理以后还是有 AI 味。**

### 核心任务

#### 任务 A：Voice Calibration Lite

给用户一个明确入口：

- “下面是我平时会写的 2–3 段文字，请按这个口吻改写”
- 不模仿字面句子
- 只拟合节奏、口语度、连接方式和判断力度

#### 任务 B：Residual Audit / Two-pass 回归

增加第二遍短审：

- 第一遍：按现有规则清理
- 第二遍：只查“残留味”和“过度工整感”

#### 任务 C：Positive Style Contract

新增正向风格文件，明确改写后的目标不是“标准答案风”，而是：

- 具体
- 直说
- 场景统一
- 稍有呼吸感
- 不端着

#### 任务 D：Protected Spans / Facts Ledger

把不该动的内容先抽出来，提高稳定性，尤其是：

- docs
- status
- code-context
- mixed 场景

#### 任务 E：真实样本包

至少补 20 条真实工作流样本：

- README 简介
- GitHub release
- X 短帖
- Linux.do 长帖
- commit message
- docstring
- issue 回复
- 技术总结

### 验收标准

- 真实样本对比里，“更像人”的主观偏好显著高于当前版本
- 不明显增加误杀
- 用户提供 voice 样本后，结果的“像本人”感知有提升
- 改写后“可直接发”的比例提升

## 8.2 v1.8：从协议到 workflow

### 目标

让它更适合长期集成到 Codex / Claude Code / ChatGPT Project 里。

### 核心任务

- 场景包细分：`forum-post / X-post / README / release-note / commit-message / docstring`
- 结构化输出 schema，方便 agent 调用
- eval 2.0：新增“像不像真人 / 是否可直接发 / 是否像用户本人”三类评测
- examples 升级：加入“同一段文本在不同场景下的改法”
- 安装路径升级：清楚区分 `lite` 和 `full` 模式

## 8.3 v2.0：从仓库到小品牌

### 目标

让 `说人话` 从单一 skill 仓库，变成一个有辨识度的中文开发者工具项目。

### 核心任务

- 更标准的 README 与示例页
- 面向多平台的安装文档
- bad-case 征集机制
- 外部目录收录
- 发布节奏和社区协作机制

## 9. 指标体系

## 9.1 质量指标

### 规则层

- Should Fix 通过率：> 95%
- Should Not Fix 误杀率：< 5%

### 体验层

新增人工评测维度：

- 自然度
- 具体度
- 语域一致
- 可直接发布性
- 像本人程度（在提供样本时）

建议 5 分制。

## 9.2 迭代指标

- 每个版本新增多少真实样本
- 每个版本新增多少边界案例
- 每个版本减少了多少“清理后仍像 AI”的 bad case

## 9.3 分发指标

不要只看 star。更重要的是：

- 有多少真实 bad cases 被提交
- 有多少外部目录收录
- 有多少用户复用到自己的 agent / editor 里
- 有多少人拿 README / examples 就能上手

## 10. 评测体系升级建议

## 10.1 新增两套评测

### 套件 A：Human Preference Eval

给同一段文本：

- 原始版本
- 当前版本改写
- 新版本改写

让评测只回答：

- 哪个更像真人
- 哪个更适合直接发
- 哪个更像作者自己

### 套件 B：Voice Match Eval

前置提供用户样本，再给待改文本，评估：

- 语气是否接近
- 句长是否接近
- 是否保留了作者原有表达习惯

## 10.2 评分 rubric 建议

每项 1–5 分：

1. **保真**：事实、术语、责任主体是否完整
2. **自然**：是否仍像模型模板
3. **具体**：是否还是抽象大词
4. **语域一致**：是否混搭
5. **可发**：是否可以直接贴到真实场景
6. **像本人**：仅在给出样本时评估

## 11. 风险与边界

## 11.1 最大风险

### 风险 1：越改越平

如果只加“不要……不要……”，结果会越来越干净，但也越来越无味。

### 风险 2：为了自然感引入编造

如果没有 protected spans / facts ledger，模型可能为了自然而补细节，这是必须避免的。

### 风险 3：voice calibration 被误用成模仿名人

必须明确只允许拟合用户自己提供的样本，不做名人、品牌、公众人物模仿。

### 风险 4：为了增长去碰“骗 detector”叙事

这会把项目带偏，而且会削弱它作为 serious tool 的可信度。

## 11.2 明确边界声明

`说人话` 的目标是：

- 降低模板感
- 提高自然表达
- 守住事实与场景边界

不是：

- 伪装来源
- 伪造人设
- 规避审核
- 伪装为真人写作证据

## 14. 最后的产品判断

这个项目 **值得继续做**，而且方向是对的。

但下一阶段不能再把主要精力放在“继续加新词”上了。真正会把上限拉开的，是下面这条主线：

> **从“规则清理器”升级为“保真前提下的自然表达协议”。**

具体落地，就是 4 个关键词：

- **voice**
- **residual audit**
- **protected spans**
- **real-sample eval**

把这 4 个补起来，`说人话` 才会从“有梗、有洞察的练手项目”真正进入“可持续迭代的产品阶段”。
